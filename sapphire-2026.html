Run — two vectors, one outcome

How it happened.

2021 — Credentials stolen
Infostealer malware harvests Jira credentials. They sit in attacker hands for four years. Never rotated.
March 2025 — Reconnaissance
Hellcat ransomware group uses stolen Jira credentials to map the internal network. Host files, DNS entries, SAP system landscape. They know what's running and where.
April 2025 — SAP releases a patch
CVE-2025-31324 — CVSS 10.0. Unauthenticated file upload in SAP NetWeaver Visual Composer. SAP publishes the fix. JLR doesn't apply it.
August 2025 — The kill shot
Attackers exploit the unpatched endpoint. HTTP POST to /irj/portal → webshell deployed → full system access. No credentials needed. No phishing. Just an open door.
September 2025 — Total shutdown
Three UK plants offline. 350GB exfiltrated. 5,000+ supply chain organizations disrupted. £20 million per day in losses.

Run — the foundation

It starts with standardization.

Red Hat Enterprise Linux is SAP's preferred OS for net new RISE deployments. 27 years of co-engineering. Extended Update Support aligned to SAP's release lifecycle.

RHEL 10 was certified for SAP HANA in February 2026 — SAP supports that combination through 2035. That's the longest SAP HANA support window available on any operating system today. When you're planning a platform that needs to run for a decade, the lifecycle math matters.

Nearly 20 compliance profiles ship in the box: DISA STIG, CIS, NIST 800-53, PCI-DSS. No additional purchase. Standardize on RHEL and you get a defensible, auditable, automatable foundation. Stop managing snowflakes. Start managing policy.

Simplify — automation that closes the loop

The platform isn't enough.

A hardened OS is table stakes. What breaks SAP operations isn't the platform — it's the manual work around the platform. Patching cycles that slip. Drift that accumulates. Compliance checks that happen quarterly instead of continuously.

The question isn't whether you have the tools. It's whether the tools run themselves.

Simplify — the operational model

Day 0. Day 1. Day 2+.

0

2+

Expand — what comes next

Clean core stays clean.

PI/PO is retiring. SAP's clean core strategy means custom code moves out of the ERP. Middleware gets replaced. New workloads need a home that isn't inside the core.

Red Hat OpenShift runs SAP BTP side-by-side extensions. Hosts SAP Edge Integration Cell. Replaces PI/PO with containerized integration. Innovation happens on the platform, not in the ERP — and it runs on the same certified, hardened RHEL foundation you already trust.

Expand — beyond the ERP

IT/OT convergence.

SAP is one source system. Your factories, warehouses, hospitals, and retail locations have hundreds more — devices, sensors, building management, edge compute. The data flows in both directions.

OpenShift is source-agnostic. SAP connects through EIC and BTP. Other systems connect natively — MQTT for devices, REST for SaaS, OPC-UA for industrial. The platform doesn't care what the source is. It cares that every connection is secured, observed, and automated.

The same architecture from core to edge. One platform for everything that touches your business.

Expand — intelligence at the edge

Not every event needs to go upstream.

A smart platform reduces noise before it reaches SAP. Three layers of intelligence, each optional, each additive:

Event-Driven Ansible

Deterministic. Rules, batching, dedup, throttle. Known patterns get handled automatically. Already running on your cluster — it's part of AAP.

Red Hat OpenShift AI

Nondeterministic. Anomaly detection, event consolidation, pattern recognition. The things rules can't catch.

IBM Granite

Local-first inference. Triage, classification, summarization — on-premises, air-gapped if needed. No data leaves the building.

Only what matters goes upstream. Transaction volume drops. Signal-to-noise improves. SAP gets cleaner data, not more data.
